Lab 2: Simplex do MILP w Pyomo

Pytanie przewodnie

Co zyskujemy, gdy zamiast reguły zachłannej zapisujemy problem jako model matematyczny?

Cel laboratorium

Po tym labie student powinien rozumieć:

  1. Różnicę między relaksacją LP a modelem binarnym.
  2. Dlaczego rozwiązanie ułamkowe może być świetną granicą, ale złym planem operacyjnym.
  3. Jak Pyomo oddziela model od solvera.

1. Minimalny kontrprzykład

Wracamy do instancji z labu 1. Greedy wybiera A, ale model binarny powinien wskazać B + C.

2. LP-relaxation

W relaksacji LP zmienna x_i może przyjąć wartość z przedziału [0, 1]. To odpowiada geometrii simpleksu: poruszamy się po wielościanie, ale nie wymagamy jeszcze decyzji całkowitych.

lp_model = build_pyomo_outage_model(tasks, limits, binary=False)
lp_result = solve_pyomo_model(lp_model)

pd.DataFrame(lp_result["variables"])
task_id x
0 A 1.0
1 B 0.0
2 C 0.8 
{
    "status": lp_result["termination_condition"],
    "objective": lp_result["objective"],
    "selected": lp_result["selected_task_ids"],
}
{'status': 'optimal', 'objective': 15.600000000000001, 'selected': ['A', 'C']}

3. MILP

Teraz narzucamy x_i ∈ {0, 1}. To jest już decyzja: praca jest wykonana albo nie.

milp_result = solve_pyomo_model(milp_model)

pd.DataFrame(milp_result["variables"])

Interpretacja

Pyomo nie jest „algorytmem simpleks”. Jest językiem modelowania. Solver dostaje symboliczną strukturę: zmienne, ograniczenia i funkcję celu. Dzięki temu ten sam model można potraktować jako LP-relaksację albo MILP.

Zadania

  1. Zmień limit dawki i sprawdź, które prace wypadają z rozwiązania.
  2. Dodaj ograniczenie: maksymalnie 10 wybranych prac.
  3. Dodaj drugą funkcję celu przez ważoną karę za dawkę.
